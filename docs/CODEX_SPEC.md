# Спецификация проекта для Codex
**Проект:** Информационная система для сети вендинговых аппаратов  
**Версия:** 1.0  
**Язык интерфейса:** русский  
**Назначение документа:** этот файл нужен как основная проектная спецификация для Codex/разработчика. Он агрегирует требования из исходного ТЗ и переводит их в структуру, пригодную для реализации.

---

## 1. Контекст и цель проекта

Нужно реализовать информационную систему для сети вендинговых аппаратов. Система должна закрывать четыре блока задач:

1. управление сетью торговых автоматов;
2. учет продаж, товаров и обслуживания;
3. личный кабинет франчайзера в формате desktop-приложения;
4. веб-модуль «Монитор ТА» для подробного мониторинга состояния автоматов.

Также проект должен включать:
- реляционную БД, приведенную к 3НФ;
- импорт/загрузку стартовых тестовых данных;
- ручные UI тест-кейсы;
- автоматизированные UNIT-тесты для авторизации и регистрации.

---

## 2. Границы проекта

### Входит в scope
- проектирование схемы БД;
- создание миграций / SQL-скриптов / DDL;
- загрузка стартовых данных;
- реализация desktop-модуля франчайзера;
- реализация web-модуля «Монитор ТА»;
- реализация регистрации, авторизации и связанных проверок;
- реализация базовой аналитики и отчетных блоков;
- экспорт списков в CSV;
- 10 ручных тест-кейсов;
- не менее 15 UNIT-тестов для процессов регистрации и авторизации;
- документация по запуску проекта.

### Не входит в обязательный scope
- реальная отправка email через SMTP;
- полноценная внешняя биллинговая интеграция;
- мобильное приложение;
- реальная оплата товаров через аппарат;
- сложная BI-аналитика за пределами блоков из ТЗ.

## 6. Функциональные требования — база данных

## 6.1 Общие требования к БД
**MUST**
- Использовать реляционную СУБД: MySQL / Microsoft SQL Server / PostgreSQL / РедБД.
- Привести схему минимум к 3НФ.
- Создать таблицы основных сущностей.
- Настроить PK, FK, UNIQUE, CHECK, NOT NULL там, где это необходимо.
- Импортировать стартовые данные из предоставленного набора данных или обеспечить эквивалентный seed.
- После разворачивания БД в таблицах должны быть тестовые данные, достаточные для демонстрации всех модулей.

**SHOULD**
- Использовать миграции.
- Использовать справочники вместо «магических строк».
- Хранить даты в UTC либо явно хранить timezone-aware значения, а в UI конвертировать в нужный часовой пояс.

## 7. Бизнес-правила и валидации

## 7.1 Регистрация франчайзера
**MUST**
1. Пользователь вводит email и пароль.
2. Система валидирует email.
3. Система валидирует пароль:
    - минимум 8 символов;
    - минимум 1 цифра;
    - минимум 1 спецсимвол.
4. Пользователь проходит математическую CAPTCHA:
    - выражение содержит минимум 3 арифметические операции.
5. Система генерирует код подтверждения email.
6. Для эмуляции код показывается в модальном окне.
7. Пользователь вводит код подтверждения.
8. Пользователь вводит уникальный код франчайзи.
9. После успешной проверки кода подтверждения и кода франчайзи создается учетная запись.
10. Пользователь получает доступ в личный кабинет.

**SHOULD**
- код подтверждения иметь ограничение по времени жизни;
- число попыток ввода кода ограничивать;
- пароль хранить только в хеше.

## 7.2 Авторизация
**MUST**
- вход по email + пароль;
- при успешном входе открывается «Главная»;
- при ошибке показывается понятное сообщение;
- должна быть возможность показать/скрыть пароль.

## 7.3 Общие правила по данным
**MUST**
- email пользователя уникален;
- компания может иметь вышестоящую компанию;
- торговый автомат принадлежит компании;
- продажа всегда привязана к автомату и товару;
- запись обслуживания всегда привязана к автомату;
- при удалении связанных сущностей не допускать нарушения ссылочной целостности.

## 7.4 Отвязка модема
ТЗ требует, чтобы после отвязки поле модема в карточке автомата принимало значение `-1`.

**Реализация для Codex:**
- в БД хранить `modem_id = NULL` как корректное отношение;
- в UI, CSV и DTO для поля «Модем» отображать `-1`, если модем отвязан.

## 7.5 Удаление сущностей
**MUST**
- удаление торгового автомата должно требовать подтверждение пользователя;
- удаление компании должно требовать подтверждение пользователя;
- при невозможности физического удаления из-за связей допускается мягкое удаление (`is_deleted`) или деактивация, но поведение в UI должно соответствовать действию «Удалить».

- кнопка входа;

### 8.2.2 Экран регистрации
**MUST**
- поле email;

- код подтверждения сгенерирован;
- неверный код подтверждения;
- неверный код франчайзи;
- успешная регистрация.

## 9. Главная страница desktop-приложения

На главной странице должны быть 5 информационных блоков.

## 9.1 Блок «Эффективность сети»
**MUST**
- отображать процент работающих автоматов по всей сети.

**Формула:**
`работающие автоматы / все автоматы * 100`

## 9.2 Блок «Состояние сети»
**MUST**
- отображать диаграмму по статусам:
    - работает;
    - не работает;
    - на обслуживании.

## 9.3 Блок «Сводка»
**MUST**
- отображать агрегированные данные по:
    - продажам;
    - инкассации;
    - обслуживанию ТА.

**Рекомендуемый состав показателей:**
- денег в ТА;
- сдача в ТА;
- выручка сегодня;
- выручка вчера;
- инкассировано сегодня;
- инкассировано вчера;
- обслужено ТА сегодня/вчера.

## 9.4 Блок «Динамика продаж за последние 10 дней»
**MUST**
- отображать график за последние 10 дней;
- поддерживать переключение:
    - по сумме;
    - по количеству.

**Правило агрегации:**
- группировка по календарным дням;
- диапазон — последние 10 завершенных или включающих текущий день календарных суток (допустимо выбрать один вариант, но зафиксировать в коде и README).

## 9.5 Блок «Новости»
**MUST**
- отображать список новостей франчайзера.

**Допущение**
- если внешний источник новостей не предоставлен, использовать таблицу `news_items` и seed-данные.

- при создании автомата должна быть выбрана компания/владелец через связанный контекст, если это требуется архитектурой.

### 10.2.3 Поведение при отвязке модема
**MUST**
- показать подтверждение;
- показать уведомление об успешной операции;
- автомат остается в списке;
- поле «Модем» после отвязки отображает `-1`.

## 18. Неоднозначности ТЗ и принятые допущения

1. **Стек не задан.**

6. **Дополнительные статусы монитора заданы визуально, но не расшифрованы текстом.**
