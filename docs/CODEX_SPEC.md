# Спецификация проекта Automata

**Проект:** информационная система для сети вендинговых аппаратов.  
**Версия:** 2.1  
**Язык интерфейса:** русский.

## 1. Основной принцип

Проект должен быть **максимально простым**. Нельзя добавлять таблицы, сущности и сценарии, которые не нужны для базовой работы приложения.

Главный ориентир для текущей итерации — минимальная рабочая БД и простая предметная модель без лишних справочников и промежуточных таблиц.

## 2. Уточнение по авторизации и регистрации

Для текущей версии проекта:
- подтверждение email **не нужно**;
- отдельные таблицы для кодов подтверждения email **не нужны**;
- дополнительные сущности для сложной регистрации **не нужны**;
- пароль пользователя хранится только в виде хеша.

Если позже логика регистрации будет расширяться, это нужно делать отдельным изменением, а не закладывать заранее в БД.

## 3. Требование к базе данных

База данных должна быть создана в схеме `automata`.

База данных должна содержать **только следующие таблицы**:
1. `vending_machines` — вендинговые аппараты;
2. `products` — товары;
3. `sales` — продажи;
4. `users` — пользователи;
5. `maintenance_records` — обслуживание;
6. `machine_statuses` — статус аппарата;
7. `machine_models` — модель аппарата;
8. `roles` — роль пользователя.

Других таблиц в текущей версии БД быть не должно.

## 4. Минимальная логическая модель

### 4.1 `roles`
Хранит роли пользователей.

Поля:
- `id`
- `name`

### 4.2 `machine_statuses`
Хранит статусы аппаратов.

Минимальные значения:
- `Рабочий`
- `Не рабочий`

Поля:
- `id`
- `name`

### 4.3 `machine_models`
Хранит модели аппаратов.

Поля:
- `id`
- `brand`
- `model_name`

### 4.4 `users`
Хранит пользователей системы.

Поля:
- `id`
- `last_name`
- `first_name`
- `middle_name`
- `email`
- `phone`
- `password_hash`
- `role_id`
- `created_at`

### 4.5 `vending_machines`
Хранит автоматы.

Поля:
- `id`
- `name`
- `location`
- `machine_model_id`
- `status_id`
- `installed_at`
- `last_service_at`
- `total_income`

### 4.6 `products`
Хранит товары, привязанные к автомату.

Поля:
- `id`
- `machine_id`
- `name`
- `description`
- `price`
- `quantity`
- `min_stock`
- `avg_daily_sales`

### 4.7 `sales`
Хранит продажи.

Поля:
- `id`
- `machine_id`
- `product_id`
- `quantity`
- `sale_amount`
- `sale_datetime`
- `payment_method`

Метод оплаты хранится строкой в самой таблице продаж, без отдельного справочника.

### 4.8 `maintenance_records`
Хранит обслуживание аппаратов.

Поля:
- `id`
- `machine_id`
- `user_id`
- `service_date`
- `work_description`
- `issues`

## 5. Ограничения

Обязательно:
- использовать PK и FK только там, где они действительно нужны;
- не добавлять таблицы для email-подтверждения, франчайзи-кодов, модемов, компаний, оборудования, событий, новостей и иных расширений;
- не дробить оплату, остатки и статусы на дополнительные сущности, если это не требуется напрямую;
- сохранить 3НФ в разумном и простом виде.

## 6. Seed-данные

В БД должны быть стартовые данные:
- роли пользователей;
- статусы аппаратов;
- модели аппаратов;
- пользователи;
- автоматы;
- товары;
- продажи;
- записи обслуживания.

Данных должно хватать для демонстрации основных экранов и запросов.
