# Спецификация проекта Automata

**Проект:** информационная система для сети вендинговых аппаратов.  
**Актуальная версия:** 2.0  
**Язык интерфейса:** русский.  
**Основные источники требований:**
1. `ЗАдание УП01 2026.docx` — основное ТЗ.
2. Макеты и визуальные референсы в `assets/`.
3. Черновой ER-набросок в `docs/er/ER-excalidraw.excalidraw`.

Этот документ заменяет прежнюю расширенную спецификацию и фиксирует минимально необходимый, но полный объём требований для дальнейшей разработки. Его задача — удерживать проект в рамках ТЗ, не раздувать модель данных и не добавлять сущности, которых нет в предметной области задания.

---

## 1. Цель проекта

Нужно реализовать систему, которая покрывает три практические задачи:

1. хранение и обработку данных по сети вендинговых аппаратов;
2. desktop-кабинет франчайзера для входа, регистрации и администрирования;
3. web-модуль «Монитор ТА» для наблюдения за состоянием автоматов.

Система должна быть пригодна для учебной демонстрации и защиты: данные согласованы, сценарии воспроизводимы, а функциональность явно соответствует ТЗ.

---

## 2. Что входит в обязательный результат

### 2.1 База данных
Обязательно:
- реляционная БД;
- схема в 3НФ без искусственного усложнения;
- схема `automata`;
- SQL-скрипт создания структуры;
- SQL-скрипт заполнения стартовыми данными;
- ограничения целостности: PK, FK, UNIQUE, CHECK, NOT NULL там, где это действительно нужно;
- данные, достаточные для демонстрации desktop- и web-модулей.

### 2.2 Desktop-приложение
Обязательно реализовать:
- экран входа;
- экран регистрации;
- экран «Главная»;
- меню «Администрирование»;
- раздел «Торговые автоматы»;
- раздел «Компании».

### 2.3 Web-приложение
Обязательно реализовать:
- экран «Монитор ТА».

### 2.4 Тесты и документация
Обязательно:
- не менее 10 ручных тест-кейсов для desktop UI;
- не менее 15 unit-тестов для авторизации и регистрации;
- README с инструкцией запуска.

## 3. Принципы проектирования

1. **Не менять стек без крайней необходимости.**
2. **Не раздувать БД.** Если сущность не нужна для сценариев ТЗ, её не добавлять.
3. **Нормализация должна быть разумной.** Цель — 3НФ, а не академическое дробление на десятки справочников.
4. **Если текст ТЗ и макет расходятся по полям и видимым элементам — брать макет.**
5. **Все пользовательские надписи в UI должны быть на русском языке.**
6. **Пароли хранить только в виде хеша.**
7. **Реальную отправку email не реализовывать.** Код подтверждения нужно эмулировать в модальном окне.
8. **Для регистрации обязательна математическая CAPTCHA минимум с 3 арифметическими операциями.**
9. **После отвязки модема в UI и CSV показывать `-1`, даже если в БД хранится `NULL`.**
10. **Если реальный API для web-монитора недоступен, использовать mock/stub adapter с фиксированным контрактом.**
11. **Для списков обязательны фильтр, пагинация, экспорт CSV и корректный показ количества записей.**
12. **Любые деструктивные действия подтверждать модальным окном.**

---

## 4. Пользователи и роли

Минимально система должна поддерживать:
- **гость** — может открыть вход и регистрацию;
- **франчайзер / пользователь системы** — входит в ЛК и работает с функциями проекта;
- **роли в БД** минимум: `Администратор`, `Оператор`.

Дополнительные роли допустимы только если они реально используются интерфейсом или логикой.

## 5. Функциональные требования к БД

## 5.1 Общие требования
База данных должна хранить сведения по следующим предметным областям:
- компании;
- пользователи;
- коды франчайзи и коды подтверждения email;
- торговые автоматы;
- модели и типы автоматов;
- модемы, провайдеры и типы подключения;
- товары и остатки в автоматах;
- продажи;
- обслуживание;
- события автоматов;
- оборудование автоматов;
- новости для главной страницы.

## 5.2 Правило минимальной достаточности
Нельзя добавлять сущности наподобие шаблонов, матриц, каталогов и вспомогательных справочников, если без них можно корректно реализовать ТЗ в 3НФ. Каждая таблица должна быть объяснима через один из экранов, сценариев или обязательных отчётных блоков.

## 5.3 Минимальная логическая модель

- `payment_methods`

### Основные сущности

- `franchise_codes`
- `email_verification_codes`

- `modems`
- `vending_machines`
- `machine_payment_methods`

Такой набор покрывает ТЗ и ER-набросок без переусложнения.

## 6. Требования к данным по предметной области

## 6.1 Торговые автоматы
Для автомата должны храниться как минимум:
- идентификатор;
- название автомата;
- компания;
- модель;
- тип автомата;
- статус;
- адрес / место установки;
- дата установки;
- дата последнего обслуживания;
- совокупный доход;
- модем, если он привязан.

Допускается хранить способы оплаты через отдельную связующую таблицу, а не одной строкой.

## 6.2 Товары
Для товара и остатков должны быть доступны:
- идентификатор товара;
- название;
- описание;
- цена;
- количество в наличии;
- минимальный запас;
- средние продажи в день.

## 6.3 Продажи
Для продажи должны быть доступны:
- автомат;
- товар;
- количество;
- сумма продажи;
- дата и время;
- метод оплаты.

## 6.4 Пользователи
Для пользователя должны быть доступны:
- ФИО;
- email;
- телефон;
- роль;
- пароль в виде хеша;
- признак активности.

## 6.5 Обслуживание
Для записи обслуживания должны быть доступны:
- автомат;
- дата обслуживания;
- описание работ;
- проблемы;
- исполнитель.

## 6.6 Данные для монитора ТА
Из БД должны быть доступны:
- информация об автомате;
- провайдер;
- деньги на счету / накопленный доход, если это требуется экраном;
- события;
- оборудование.

Из внешнего API или mock/stub-адаптера могут приходить:
- состояние связи;
- загрузка товарами;
- денежные средства внутри автомата;
- дополнительные статусы.

## 7. Требования к desktop-модулю

## 7.1 Авторизация
Экран входа должен позволять пользователю войти в личный кабинет.

Минимум:
- email;
- пароль;
- валидация формата;
- переход на главную страницу после успешного входа.

## 7.2 Регистрация
Сценарий регистрации обязателен в следующем порядке:
1. пользователь вводит email и пароль;
2. система генерирует код подтверждения и показывает его в модальном окне;
3. пользователь вводит код подтверждения;
4. пользователь вводит код франчайзи;
5. пользователь решает CAPTCHA;
6. после успешной проверки получает доступ к личному кабинету.

Обязательные проверки:
- email валиден;
- пароль не короче 8 символов и содержит цифры и спецсимволы;
- код подтверждения проверяется;
- код франчайзи проверяется;
- CAPTCHA содержит минимум 3 арифметические операции.

## 7.3 Главная
На главной странице должны быть 5 блоков:
1. эффективность сети;
2. состояние сети;
3. сводка;
4. динамика продаж за последние 10 дней;
5. новости.

## 7.4 Администрирование → Торговые автоматы
Список должен поддерживать:
- таблицу и плитку;
- по умолчанию таблицу;
- выбор количества строк;
- фильтр по названию автомата;
- пагинацию;
- показ количества записей;
- подсветку нечётных строк;
- экспорт CSV.

Действия по автомату:
- добавить;
- редактировать;
- удалить с подтверждением;
- отвязать модем с подтверждением.

После отвязки модема:
- автомат остаётся в списке;
- в интерфейсе и CSV модем отображается как `-1`.

## 7.5 Администрирование → Компании
Список компаний должен поддерживать:
- выбор количества строк;
- фильтр по названию компании;
- выбор вышестоящей компании;
- показ количества записей;
- подсветку нечётных строк;
- экспорт CSV.

Действия:
- добавить компанию;
- редактировать;
- удалить с подтверждением.

---

## 8. Требования к web-модулю «Монитор ТА»

Экран должен поддерживать:
- фильтр по общему состоянию;
- фильтр по типу подключения;
- фильтр по дополнительным статусам;
- применение фильтров по кнопке «Применить»;
- сообщение при отсутствии результатов;
- дублирование результата фильтрации в поле «Итого автоматов»;
- пересчёт поля «Денег в автоматах» по отфильтрованной выборке;
- сортировку по состоянию ТА.

Таблица должна отображать колонки:
- №;
- ТП;
- Связь;
- Загрузка;
- Денежные средства;
- События;
- Оборудование;
- Информация;
- Доп.

---

## 9. Ограничения и анти-требования

Нельзя:
- хранить пароли в открытом виде;
- удалять обязательные проверки регистрации;
- убирать CSV-экспорт из списков;
- убирать сортировки и фильтры web-монитора;
- скрывать логику отвязки модема;
- подменять реальные тесты формальными заглушками без сценариев.

Нежелательно:
- добавлять сложные BI-модули;
- добавлять мобильный клиент;
- добавлять реальные почтовые интеграции;
- строить избыточную предметную модель, не подтверждённую ТЗ.

Порядок работ:

## 11. Что нужно помнить при дальнейшей разработке

- ТЗ требует именно учебную, демонстрационную, но рабочую систему.
- БД должна отражать предметную область напрямую, без лишних абстракций.
- Все решения по данным должны быть объяснимы через конкретный экран или сценарий.
- При спорных местах приоритет у макета по составу полей и элементов.
- `docs/er/ER-excalidraw.excalidraw` — это черновой ориентир, а не безусловная истина; если ER и ТЗ расходятся, приоритет у ТЗ.
